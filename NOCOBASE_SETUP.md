# Настройка NocoBase для анонимного прохождения тестов

## Обзор

Фронтенд поддерживает анонимное прохождение тестов по ссылке вида `/t/<token>`.
Для корректной работы необходимо настроить бэкенд (NocoBase).

---

## 1. Поле `token` в коллекции `attempts`

Если поле `token` ещё не добавлено в коллекцию `attempts`, его нужно создать:

| Параметр      | Значение                    |
|---------------|-----------------------------|
| Имя поля      | `token`                     |
| Тип           | `string` (однострочный)     |
| Уникальное    | **Да** (`unique: true`)     |
| Обязательное  | Нет (заполняется только для анонимных попыток) |
| Индекс        | Рекомендуется добавить      |

### Генерация токена

Токен генерируется при создании анонимной попытки. Рекомендуемый формат — UUID v4 или nanoid (21 символ).
Можно генерировать токен:
- **В NocoBase Workflow** — при создании записи `attempt` через триггер "After create" → заполнить поле `token` случайным значением.
- **Вручную** — специалист создаёт attempt через интерфейс NocoBase и копирует/генерирует токен.

---

## 2. Публичный доступ к API (ACL / Roles)

Анонимный пользователь **не аутентифицирован**. Необходимо разрешить публичный доступ (роль `public`) к следующим ресурсам:

### Чтение (list / get)

| Коллекция               | Действие          | Фильтр / Scope                   |
|--------------------------|--------------------|-----------------------------------|
| `attempts`               | `list`             | Только по `filter[token]=<value>` |
| `tests`                  | `get`              | Только `is_public = true` (если есть такое поле) |
| `questions`              | `list`             | По `test_id`                      |
| `options`                | `list`             | По `question_id`                  |
| `scales`                 | `list`             | По `id`                           |
| `scale_options`          | `list`             | По `scale_id`                     |
| `answers`                | `list`             | По `attempt_id`                   |
| `answer_ranking_items`   | `list`             | По `answer_id`                    |

### Запись (create / update / destroy)

| Коллекция               | Действие            | Примечание                                       |
|--------------------------|---------------------|--------------------------------------------------|
| `persons`                | `create`            | Создание анонимной персоны (ФИО)                 |
| `attempts`               | `update`            | Обновление `person_id`, `status`, `started_at`, `submitted_at`, `duration` |
| `answers`                | `update`            | Сохранение ответов                               |
| `answers`                | `set` (M2M/`options`) | Для `multiple_choice` — связь ответа с вариантами |
| `answer_ranking_items`   | `create`, `update`, `destroy` | Управление ранжированием                |

### Настройка в NocoBase UI

1. Перейти в **Settings → Roles & Permissions**
2. Открыть роль **Public**
3. Для каждой коллекции из таблиц выше включить соответствующие действия
4. (Рекомендуется) Настроить **Scope** для ограничения доступа:
   - `attempts`: добавить scope `token is not null` — чтобы публичный доступ был только к попыткам с токеном
   - `answers`: добавить scope на связь с `attempt.token is not null`

> ⚠️ **Безопасность**: без scope-ограничений публичная роль получит доступ ко всем записям. Обязательно настройте scope.

## 3. Создание анонимной попытки (Workflow / Вручную)

### Вариант A: Вручную через интерфейс NocoBase

1. Специалист открывает коллекцию `attempts`
2. Создаёт запись:
   - `test_id` — ID нужного теста
   - `status` — `assigned`
   - `token` — сгенерированный UUID (можно через формулу или вручную)
   - `person_id` — **оставить пустым** (заполнится при прохождении)
3. Копирует ссылку: `https://<domain>/t/<token>`
4. Отправляет ссылку респонденту

### Вариант B: Workflow для автоматической генерации

1. Создать workflow: **Trigger** → Manual (или через кнопку на форме)
2. Шаги:
   1. **Create record** в `attempts`:
      - `test_id` = входной параметр
      - `status` = `assigned`
      - `token` = `{{$system.uuid}}` или кастомная формула
   2. (Опционально) Создать записи `answers` для всех вопросов теста
   3. Вернуть `token` для формирования ссылки

### Вариант C: Автосоздание ответов

Если ответы (`answers`) создаются заранее (при создании attempt), добавить workflow:

**Trigger**: After create on `attempts`
**Condition**: `token` is not null
**Action**: Loop по `questions` с `test_id = attempt.test_id`, для каждого → create `answer` с `attempt_id` и `question_id`

> Если ответы НЕ создаются заранее, фронтенд ожидает, что они уже существуют. Необходимо либо создавать их в workflow, либо добавить логику создания на фронтенде.

---

## 4. Предварительное создание ответов (answers)

Текущая логика фронтенда **не создаёт** записи `answers` — она ожидает, что они уже существуют для каждого вопроса теста.

Варианты решения:

### A. Создавать answers в NocoBase Workflow (рекомендуется)

При создании `attempt` автоматически создавать `answer` для каждого активного `question` теста:

```
Trigger: After create → attempts (where token is not null)
Action:
  1. Query: questions where test_id = {{$context.test_id}} AND is_active = true
  2. Loop:
     - Create answer: { attempt_id: {{$context.id}}, question_id: {{$loop.id}} }
```

### B. Создавать answers на фронтенде

Добавить в `AnonAttemptView.vue` логику: если для какого-то вопроса нет answer, создать его через `create('answers', { attempt_id, question_id })`.

---

## 5. Поле `middle_name` в коллекции `persons`

Для корректного ввода ФИО убедитесь, что в коллекции `persons` есть поле:

| Параметр  | Значение              |
|-----------|-----------------------|
| Имя поля  | `middle_name`         |
| Тип       | `string` (однострочный) |
| Обязательное | Нет               |

Если поле отсутствует — добавьте его через конструктор коллекций.

---

## 6. Проверка настройки

### Чек-лист

- [ ] Поле `token` добавлено в `attempts` (unique, indexed)
- [ ] Поле `middle_name` добавлено в `persons` (если отсутствовало)
- [ ] Роль `public` имеет доступ на чтение к: `attempts`, `tests`, `questions`, `options`, `scales`, `scale_options`, `answers`, `answer_ranking_items`
- [ ] Роль `public` имеет доступ на запись к: `persons` (create), `attempts` (update), `answers` (update + set), `answer_ranking_items` (create/update/destroy)
- [ ] Scope настроены для ограничения публичного доступа (token is not null)
- [ ] Answers создаются при создании attempt (workflow или вручную)
- [ ] Тестовая ссылка `/t/<token>` открывается в браузере без авторизации
- [ ] Форма ФИО работает, person создаётся, attempt обновляется
- [ ] Ответы сохраняются, тест завершается

---

## 7. Формат ссылки

```
https://<домен>/t/<token>
```

Пример: `https://flow.skeducator.ru/t/a1b2c3d4-e5f6-7890-abcd-ef1234567890`
